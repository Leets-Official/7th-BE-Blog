## 01. 서버란 무엇인가

### 서버의 정의

> 서버란 클라이언트의 요청을 받아 처리하고, 그 결과를 다시 돌려주는 시스템

- 서버는 단순히 데이터를 저장하는 곳이 아니라, 서비스의 기능을 공통 규칙과 중앙 관리 하에 제공하는 핵심 주체다.

### 레스토랑 비유

| 레스토랑     | 웹 서비스         |
|------------|-----------------|
| 손님         | 사용자            |
| 직원/주문 창구 | 클라이언트 (웹/앱)  |
| 주방         | 서버              |
| 재고/주문 기록 | 데이터베이스 (DB)   |
| 주문         | 요청 (Request)    |
| 음식 제공     | 응답 (Response)   |

### 서버가 필요한 이유

서버 없이 모든 사용자가 각자의 기기에서 처리하면 발생하는 문제:
- 데이터가 기기마다 달라짐 (일관성 없음)
- 보안 규칙 적용 불가
- 여러 사용자 간 상태 조율 불가

서버가 있을 때:
- 여러 사용자의 요청을 안정적으로 관리
- 데이터를 일관되게 저장
- 정해진 규칙에 따라 기능 수행
- 서비스 기능을 중앙에서 제공

### 클라이언트 vs 서버

| 구분       | 역할                              |
|----------|-----------------------------------|
| 클라이언트 | 요청을 보내는 쪽 (앱, 웹 브라우저, 프론트엔드) |
| 서버       | 요청을 받아 처리하고 응답하는 쪽 (백엔드)     |

## 02. 요청(Request)과 응답(Response)

### 요청(Request)

클라이언트가 서버에게 원하는 작업을 전달하는 것. 단순한 버튼 클릭이 아니라, 여러 정보를 담아 서버로 전달되는 메시지.

요청에는 다음 정보가 포함된다:
- **어디로** 보내는지 (URL/엔드포인트)
- **어떤 방식으로** 보내는지 (HTTP Method: GET, POST 등)
- **어떤 데이터와 함께** 보내는지 (Header, Body)

**GET 요청 예시** - 메뉴 목록 조회:
```bash
curl -X GET "https://api.restaurant.com/menus" \
  -H "Accept: application/json"
```

**POST 요청 예시** - 주문 생성:
```bash
curl -X POST "https://api.restaurant.com/orders" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer example-token" \
  -d '{ "menuName": "김치볶음밥", "quantity": 2, "requestNote": "덜 맵게 해주세요" }'
```

### 응답(Response)

서버가 요청을 처리한 결과를 클라이언트에게 돌려주는 것.

**성공 응답 (201 Created)**:
```json
HTTP/1.1 201 Created
Content-Type: application/json

{
  "orderId": 1024,
  "status": "CREATED",
  "message": "김치볶음밥 주문이 완료되었습니다."
}
```

**실패 응답 (400 Bad Request)**:
```json
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": "INVALID_REQUEST",
  "message": "올바른 주문 정보가 필요합니다."
}
```

### 전체 흐름

```
사용자 (버튼 클릭/값 입력)
  -> 클라이언트 (요청 전송)
    -> 서버 (로직 처리)
      -> 데이터베이스 (데이터 조회/저장)
      <- 데이터베이스 (처리 결과 반환)
    <- 서버 (응답 전송)
  <- 클라이언트 (화면에 결과 표시)
사용자
```

## 03. Java & JVM

### Java가 서버 개발에 많이 사용되는 이유

- 비교적 안정적으로 동작
- 객체지향적으로 구조를 나누기 좋음
- 대규모 프로젝트에서 유지보수에 유리
- 다양한 라이브러리와 프레임워크 생태계

### Java의 실행 흐름

Java 코드는 바로 실행되지 않고 중간 단계를 거친다:

```
Java 코드 작성 -> 컴파일 -> 바이트코드 생성 -> JVM 실행
```

### JDK / JRE / JVM 구분

| 개념 | 역할 | 레스토랑 비유 |
|-----|------|------------|
| JVM | Java 바이트코드를 실행하는 가상 머신 | 실제로 요리를 수행하는 주방 설비 |
| JRE | Java 프로그램을 실행하기 위한 환경 | 요리가 돌아가기 위한 주방 환경 |
| JDK | Java 프로그램을 개발하기 위한 도구까지 포함한 키트 | 요리 도구와 레시피 작성 도구까지 모두 포함한 전체 세트 |

> 개발할 때는 JDK를 설치하면 JRE, JVM이 모두 포함된다.

### 가장 작은 Java 코드 예시

```java
public class Main {
    public static void main(String[] args) {
        System.out.println("안녕하세요, 서버의 세계에 오신 것을 환영합니다.");
    }
}
```

Java 프로그램은 반드시 클래스 안에 코드를 작성하는 구조를 가진다.

## 04. Spring Boot 소개

### 왜 Spring Boot인가

서버가 클라이언트의 요청을 처리하는 시스템이라는 것을 알았다면, 실제로 이 서버를 어떤 기술로 만드는지가 중요하다.

- 현대 서버 개발에서 **Java + Spring Boot** 조합은 가장 널리 사용되는 조합 중 하나
- Spring Boot는 서버 개발에 필요한 복잡한 설정을 간소화하고, 레이어드 아키텍처 기반으로 구조적인 개발을 가능하게 함

### 레이어드 아키텍처 (계층형 아키텍처)

서버 코드는 역할별로 계층을 나눠 관리한다:

```
Controller (클라이언트의 요청을 받는 계층)
    |
Service (비즈니스 로직을 처리하는 계층)
    |
Repository (DB와 통신하는 계층)
```
